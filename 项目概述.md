# 不牛马厨房｜项目概述

> 用途：帮助 AI 快速理解项目。功能设计、产品结构、核心交互、业务规则、项目定位、架构或目录变化时必须同步更新。

## 项目介绍

“不牛马厨房”是一个手机优先的做饭决策网站，核心问题是“今晚吃什么”。用户可通过转盘抽菜，并结合人数、辣度、偏好、忌口、健康模式和现有食材筛选结果，随后查看做法、管理购物清单和记录最近做过的菜。

## 产品结构

- `首页`：展示抽菜结果、人数及筛选入口；首次未抽菜时提供轻量转盘引导。整体采用手机端餐饮编辑式单列排版，以菜品照片、菜名和主操作为第一视觉层级；多人菜单预览位于主图上方，可点选卡片或使用主图左右按钮切换菜色，图片、营养、标签、食材与做法会同步更新。
- `菜库`：搜索和分类浏览菜品，可选菜或加入清单。
- `食材`：冰箱食材匹配与购物清单。
- `我的`：角色/主题装扮、宠物主题联动、健康模式和添加自定义菜。
- `全屏转盘`：首次打开显示，也可从首页进入；页内可直接切换 1、2、3、4+ 人并同步抽取道数，包含转盘减速、宠物完成动画和结果回填。转盘使用沉浸式手机全屏布局，完成后自动释放页面滚动锁并将焦点返回入口。

## 技术与目录

- 形态：无构建工具的静态网站，使用 HTML、内联 CSS 和原生 JavaScript；菜谱做法独立维护在 `site/assets/recipe-methods.js`，避免与页面交互代码混在一起。
- 主文件：`site/bu-niu-ma-kitchen.html`
- 根目录 `index.html`：GitHub Pages 与自定义域名入口，自动跳转到 `site/bu-niu-ma-kitchen.html`。
- 正式访问域名：`https://zhechun.space`；域名注册商为 Spaceship，权威 DNS 与边缘缓存由 Cloudflare Free 托管，静态站点由 GitHub Pages 发布。根域名使用 GitHub Pages 的四条 A 记录，`www` 使用 CNAME 指向 `voow1.github.io`；Cloudflare 接管后对上述记录启用代理和 CDN 缓存。
- 素材目录：`site/assets/`；菜图作者与许可记录位于 `site/assets/dishes/ATTRIBUTION.md`，可用图库、许可边界、视觉筛选和压缩规范位于 `site/assets/dishes/SOURCE_POLICY.md`。
- 项目协作文件：根目录 `AGENTS.md`、`项目概述.md`、`网站进度表.md` 均随 Git 仓库发布；AI 开始任务前必须先读取后两者，并遵循 `AGENTS.md` 的约束。
- 正式维护范围：仅当前工作区的 `site`；不要修改旧 `Codex/.../outputs` 路径。
- 页面最大宽度 620px，采用语义化主题色令牌、本地“乐米小奶泡体”和 4/8px 间距节奏；内置青柠绿、奶油杏、番茄红、莓果粉、海盐蓝、紫薯紫六套配色。主操作始终使用当前主题强调色，次操作使用强调色浅底，深色仅承担正文和标题，不作为脱离主题的黑色按钮。360–430px 为主要设计与验收尺寸，宽屏仅显示居中的小程序画布，不把首页拆成桌面双栏。
- 图标依赖 Lucide CDN；本地“乐米小奶泡体”按当前 HTML 与完整 500 道菜谱文本重新裁剪为 WOFF2（约 525KB），在文档首部预加载，且不再请求 Google Fonts 外链。品牌、标题、短标签、步骤正文、关键提醒、来源与菜库步骤摘要均继承同一全站字体；新增菜谱文字时必须重新扫描完整页面与 `recipe-methods.js` 生成子集，不能把长文本单独改成系统黑体。首次访问时正文会在字体可用前保持隐藏，字体加载成功后再一次性显示，以消除系统字体切换为定制字体的跳变；若字体请求失败，正文保持隐藏，用户刷新后重试。菜图仅来自原始作品页可核验授权的具体成品图；已本地化文件保留作者、来源与许可，发布图统一为最大边 640px、质量 76 的 WebP。当前转盘候选与抽中结果优先加载、菜库首屏 6 张立即加载，其余图片接近视口后分批加载。当前选中宠物的四个动作高优先级预载，其他宠物待机缩略图接近设置区时再加载；本地字体和角色素材可离线使用。

## 核心规则

- 每轮转盘最多 8 个候选；支持点击和拖拽抽菜。
- 支持 1、2、3、4+ 人；多人菜单先按主菜、蔬菜、汤、主食和轻菜形成互补结构，再综合口味浓淡、烹饪方式、蛋白质来源、热量与辣度进行组合评分，减少同质搭配并保留少量随机性。饮品不占正餐道数，筛选过窄时优先保证组合质量。
- 多人抽中菜单后，菜单预览优先展示在主图上方；用户可点选菜色卡片或使用主图左右切换按钮浏览，图片、营养信息、标签、食材与做法始终对应当前菜色。
- 优先避开最近 3 次做过的菜，无候选时回退。
- 忌口检查菜名、标签、食材、健康提醒和做法。
- 健康模式包括术后恢复期和低碘预案，仅作产品提醒与筛选，不作为医疗建议。
- 菜图只使用原始页可核验授权的具体成品图；已本地化的实拍图可离线展示并附作者、来源页和许可，未收录或网络加载失败时显示“暂无可授权实拍图”，不使用抽象图、无关通用食物图、AI 生成图或破损图片。每道菜当前只绑定一张已核验图片，因此首页结果和菜库卡片不提供换图按钮，避免单图被切换为空。图库和许可边界统一按 `SOURCE_POLICY.md` 执行。
- 内置菜 500 道：125 道种子菜 + 375 道按 23 个分类轮询选取的扩展菜，另叠加用户自定义菜。扩展阶段先按规范化菜名去重，移除“家常版 / 夜宵版 / 轻量版”等仅修饰后缀造成的重复，保证 500 个菜名和规范化菜名均唯一。
- 500 道内置菜统一使用菜品角色、蛋白来源、烹饪方式、时长、热量、口味、菜系与场景标签；偏好筛选按 6 组、55 个标签组合过滤，多人菜单优先直接读取蛋白与烹饪方式标签控制同质搭配。
- `recipe-methods.js` 保留原 240 道人工逐菜校订条目；新增 260 道在首次解析时按菜名、食材与菜型生成至少 4 个步骤、1 条提醒和可访问来源，并固化到逐菜方法表，后续展示不重复走临时兜底。原 `note` 字段继续承载健康/口味提示，不再冒充做法。“正宗”按当地人日常会做的地道家常版本理解，不以非遗、宴席或饭店标准作为默认答案；来源优先采用本地作者和地方家庭菜谱，再用高完成度社区菜谱交叉核验。
- 默认角色为草莓甜心小厨；宠物区同时保留奶茶小狗、Astr 和包仔小厨，所有角色统一包含 `idle`、`runRight`、`runLeft`、`done` 四种状态。
- 主题区保留六套互不重复的餐饮配色；旧版浅林绿、深森绿设置会自动迁移到青柠绿。
- 宠物主题与自由配色并存：宠物主题以全局色调统一品牌氛围，将角色道具装饰、选中动画和点击反馈集中在底部四栏导航，避免正文卡片过度装饰；手动选择自由配色会退出联动。角色选择卡会轮播待机、左右跑动和完成动作；当前已完成“草莓甜心厨房”样板，其他宠物主题待后续接入。
- 交互反馈遵循手机端单手操作：底部导航、主要按钮、筛选和人数选择均保持至少 44px 触控区域；整个小程序画布在任意触点显示约 16–23px 的半透明主题色“小果冻”软圆，按钮依靠自身轻微按压缩放，不再叠加粒子或涟漪；支持设备时仅在抽菜等重要动作使用轻震动。所有新增动效均只使用透明度与变换，并在系统启用“减少动态效果”时关闭。
- 多人菜单使用横向轻量胶片条：所有卡片统一以白色为主、在局部晕开当前主题青色，不使用与页面背景近似的实色块；图片采用贴边裁切，当前菜只通过底部短柔光表达，不改变整张卡片的底色、边框或阴影，也不使用硬质侧边标记。点选卡片或通过大图左右切换时，胶片条会自动跟随当前菜，首项贴左、末项贴右、中间项尽量居中，确保当前菜完整可见。
- 主图左右切换不使用悬浮圆按钮，改为贴住图片左右边缘的半月形薄把手：奶白雾气由边缘向图片内部自然消散，只保留低饱和主题色细箭头，无边框和悬浮投影；视觉融入图片，同时保留 44×64px 触控区域与轻微按压反馈。

## 运行与验收

- 双击根目录 `启动预览.cmd`；脚本会定位 Python、启动并等待 8787 端口服务就绪，再打开 `http://127.0.0.1:8787/bu-niu-ma-kitchen.html`，失败时保留可读错误提示。
- 出现 `ERR_CONNECTION_REFUSED` 或角色图片解码为 `0×0` 时，重新运行启动脚本并刷新。
- 修改前备份主 HTML；修改后按改动范围检查抽菜、连续重抽、筛选、菜库、冰箱推荐、清单、加菜、装扮及刷新后的本地数据。界面改动至少检查 360px、390px、430px，确认无横向滚动、底栏不遮内容、可见按钮触控区域不小于 44px，并验证“减少动态效果”。
